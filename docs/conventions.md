# Collaboration Conventions

사람과 에이전트가 같은 기준으로 커밋과 PR을 작성하기 위한 최소 규칙이다.

## Commit Rules

커밋 메시지는 아래 형식을 따른다.

```text
type: 작업 내용
```

- 커밋 전에는 포함될 파일별 변경 내용을 먼저 정리한다.
- 커밋 요청 시 어떤 파일이 왜 바뀌었는지 파일 단위로 설명할 수 있어야 한다.
- Codex에게 커밋 메시지 작성을 요청할 때는 파일별 변경 요약을 함께 전달한다.

### Types

- `feat`: 새로운 기능 추가
- `fix`: 버그 수정
- `docs`: 문서 수정 (코드 변경 없음)
- `style`: 코드 포맷팅, 세미콜론 등 스타일 변경 (논리 변경 없음)
- `refactor`: 리팩토링 (기능 변화 없음)
- `test`: 테스트 관련 코드 추가/수정
- `chore`: 빌드, 패키지 매니저 설정 등 기타 작업
- `design`: CSS 등 사용자 UI 디자인 변경
- `comment`: 필요한 주석 추가 및 변경
- `rename`: 파일 혹은 폴더명 수정 또는 이동만 수행한 경우
- `remove`: 파일 삭제만 수행한 경우
- `!HOTFIX`: 긴급한 치명적 버그 수정

## Pull Request Rules

### Create Criteria

- 기능 단위 작업 완료 시 PR을 생성한다.
- 기본 대상 브랜치는 `develop`이다.

### Title Format

PR 제목은 아래 형식을 따른다.

```text
[type] #이슈번호 제목
```

예시:

```text
[feat] #10 방 생성 API 구현
```

### Body Rules

- PR 본문은 한국어로 작성한다.
- 각 섹션은 짧은 불릿 중심으로 작성한다.
- 작업 내용은 구현 사실만 간결하게 적는다.
- 리뷰 포인트는 리뷰어가 판단해야 할 질문 형태를 우선한다.
- 테스트 결과는 실제로 확인한 항목만 적는다.
- 체크리스트는 PR 작성 시점 기준으로 사실대로 표시한다.

### Body Tone Example

아래 예시는 권장 톤과 밀도를 보여주기 위한 참고 예시다.

```md
## 📌 관련 이슈

- close #123

## ✨ 작업 내용

- Room 엔티티와 Participant 엔티티 간 1:N 관계를 설정했습니다.
- 방 생성 시 고유한 6자리 URL slug를 생성하도록 구현했습니다.
- NestJS `ValidationPipe`를 적용해 입력값 검증을 추가했습니다.

## 📸 스크린샷 / 테스트 결과

- `POST /rooms` 성공 응답을 확인했습니다.

## 🔍 리뷰 포인트

- 6자리 slug 생성 방식이 중복을 충분히 방지하는지 봐주시면 좋겠습니다.
- 방 생성 과정에서 Prisma 트랜잭션 경계가 적절한지 확인 부탁드립니다.
```

## Working With Codex

- PR 생성 직전에 현재 diff, 커밋 로그, 테스트 결과를 기준으로 본문을 작성한다.
- 커밋 메시지나 PR 본문 작성 전에는 변경 파일별 요약을 먼저 정리한다.
- 파일별 요약은 `파일 경로: 변경 이유 또는 핵심 수정 내용` 형식으로 정리한다.
- Codex에게 PR 본문 작성을 요청할 때는 이슈 번호, PR 제목, 확인한 테스트 결과를 함께 전달한다.
- 별도 지시가 없으면 Codex는 이 문서의 톤 예시를 따라 PR 제목과 본문을 작성한다.
